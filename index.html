Email Automation
Automatically send, receive, and process emails using UiPath — no manual effort needed.
What is Email Automation?
Email automation in UiPath means you write a robot (bot) that can send emails, read emails, filter them, and download attachments — all without you clicking a single button. Think of it as giving your computer a brain to handle your inbox.

📡 Email Protocols — How Emails Travel

📤

SMTP

Simple Mail Transfer Protocol — used to send emails. When your bot needs to send an email from Gmail or Outlook, it uses SMTP.


Server: smtp.gmail.com  |  Port: 465 (SSL)

📥

IMAP

Internet Message Access Protocol — used to read emails from a server. IMAP keeps emails on the server so you can access them from multiple devices.


Server: imap.gmail.com  |  Port: 993

🖥️

Outlook Integration

Works directly with the Microsoft Outlook desktop application. No server config needed — just the installed app. Easiest to set up for beginners!

📋 SMTP Email — Step by Step (Send SMTP Mail)

1

Install Mail Package

In UiPath Studio → Manage Packages → search UiPath.Mail.Activities → Install.

2

Add "Send SMTP Mail Message" Activity

Drag it into your workflow sequence from the Activities panel.

3

Fill in Properties

Set: To (recipient), Subject, Body, Server (smtp.gmail.com), Port (465), Email & Password.

4

Run & Test

Click Run. Check the recipient inbox to confirm the email was delivered. Execution time is usually under 5 seconds.

File & Directory Handling
Read, write, copy, move, and delete files — and manage folders automatically.
What is File Handling?
In UiPath, your robot can work with files just like you would — opening spreadsheets, reading text files, saving documents, and organizing them into folders. This is extremely useful for data entry jobs, report generation, and batch processing.

🗂️ Directory (Folder) Operations

📁

Create Directory

Creates a new folder at a specified path. Useful for organizing processed files.

🗑️

Delete Directory

Deletes a folder and all its contents. Use carefully — this is permanent!

🔍

For Each File in Folder

Loops through every file in a folder so you can process them one by one.

🔄 File Processing Workflow

1

Get List of Files

Use For Each File in Folder to loop through a directory (e.g., Downloads folder). You can filter by extension like *.xlsx.

2

Read File Content

Use Read Text File or Read CSV to load the file data into a variable.

3

Process the Data

Apply logic — filter rows, compute values, format text, etc.

4

Write Output

Save results to a new file using Write Text File or Write CSV.

5

Move Processed Files

Use Move File to send the processed file to an "Archive" folder, keeping your workspace clean.

// Example: Reading each file in Downloads folder For Each CurrentFile In Downloads_Folder // Filter only Excel files If Path.GetExtension(CurrentFile) = ".xlsx" ReadRange(CurrentFile) → Store in DataTable // ... process data ... MoveFile(CurrentFile, ArchiveFolder) End If Next

Data Table Operations
Store and manipulate data in rows & columns — just like a spreadsheet in memory.
What is a DataTable?
A DataTable is like a temporary spreadsheet that lives in your robot's memory. It has rows and columns. You can fill it from Excel, filter its rows, add new columns, loop through it, and write it back to a file. It's one of the most important concepts in UiPath!

🏗️ How to Create a DataTable

🔨

Build Data Table Activity

Manually create a DataTable by defining column names and data types, then add rows one by one. Good for small, fixed data sets.


Example columns: Name (String), Surname (String), Grade (Int32)

📊

Read Range (Excel)

Read an entire Excel sheet into a DataTable with one activity. Check "Has Headers" so column names are recognized automatically.

📄

Read CSV

Load a CSV file into a DataTable. Perfect for processing exported data from databases or web applications.

🔁 Looping Through a DataTable

// Use "For Each Row" activity to iterate For Each row In myDataTable // Access column value using column name studentName = row("Name").ToString grade = row("Grade").ToString // Example: Check if grade is above 8 If Convert.ToInt32(grade) >= 8 Log Message: studentName + " passed!" End If Next
💡

Beginner Tip: Always use row("ColumnName").ToString to read values. The column name must match exactly as it appears in the header (case-sensitive). If you get an error, check for typos!

📋 Example Logic — Processing Pending Orders

1

Read Excel File

Use Read Range → saves data into dt_Orders DataTable variable.

2

Filter Pending Rows

Filter Data Table where Status = "Pending" → saves to dt_Pending.

3

Loop & Process

For Each Row in dt_Pending → process each order (enter into system, generate PDF, etc.)

4

Write Results

Update Status column to "Completed" and write the DataTable back to Excel.

Publishing Automation Workflows
Deploy your automation from UiPath Studio to Orchestrator so robots can run it.
What is Publishing?
When you finish building an automation in UiPath Studio, it lives only on your computer. Publishing packages your workflow and uploads it to Orchestrator — the central management platform — so that any robot (on any machine) can run it, even without you being present.

📦 Key Terms to Know

📦

Package

The published version of your workflow. It's a compressed file (like a .zip) uploaded to Orchestrator containing all your automation code.

▶️

Process

A package linked to a folder in Orchestrator. It is the executable version — what you actually run on a robot. Package + Folder = Process.

📋

Job

A single execution of a Process. Every time you run a process, Orchestrator creates a "Job" record that you can monitor and track.

🪜 Steps to Publish a Workflow

1

Design Workflow in UiPath Studio

Build and test your automation completely. Make sure it runs without errors.

2

Click "Publish" Button

In the Studio top menu → click Publish. Choose the target (Orchestrator Tenant Feed).

3

Workflow is Packaged

Studio converts your project into a .nupkg package with version number (e.g., v1.0.0).

4

Package Uploaded to Orchestrator

The package appears in Orchestrator under Automations → Packages.

5

Create a Process

In Orchestrator → go to a Folder → Add → Process → select your package. This links the package to the folder.

6

Assign Process to Robot & Run

Click Start Job to run the process on a connected robot immediately, or set a schedule.

💡

Beginner Tip: Every time you update your workflow and re-publish, Orchestrator creates a new version (1.0.1, 1.0.2, etc.). Old versions are still saved, so you can always roll back if something goes wrong.

UiPath Orchestrator
The web-based command center to manage, deploy, and monitor all your automation robots.
What is Orchestrator?
Imagine you have 100 robots working for your company. You need a way to tell them what to do, when to do it, and keep track of whether they succeeded or failed. Orchestrator is that control center — accessible from any web browser, anywhere in the world.

🌟 Key Features

🤖

Robot Management

Add, configure, and monitor all your robots (attended and unattended) from one place.

▶️

Job Execution

Start jobs manually or via triggers. See which jobs are running, queued, or completed.

🔐

Asset Management

Store credentials, settings, and shared data (called Assets) securely in Orchestrator.

📋

Queue Management

Create work queues — lists of items for robots to process one by one (like a to-do list for robots).

📊

Monitoring & Logs

View real-time dashboards of robot performance, execution history, and detailed logs.

📁

Folder Structure

Organize your processes, robots, assets, and queues into folders (like departments in a company).

Robot Configuration
Set up and connect robots to Orchestrator so they can receive and execute tasks.
What is Robot Configuration?
Before a robot can run automations from Orchestrator, you need to register the machine and user in Orchestrator, then connect UiPath Assistant on that machine using a Machine Key. Think of it as activating a phone plan for your robot.

🤖 Types of Robots

👤🤖

Attended Robot

Works alongside a human. Triggered by the user when needed. Cannot run unattended.

  • Runs on the user's active desktop
  • Started from UiPath Assistant
  • Human supervises each run
  • Best for: Call center, data entry with human decisions
🤖⚙️

Unattended Robot

Runs completely on its own without any human. Triggered from Orchestrator by schedule or queue.

  • Runs 24/7 without human presence
  • Started remotely from Orchestrator
  • Needs dedicated machine/VM
  • Best for: Invoice processing, data migration, reporting

🔌 Steps to Configure a Robot

1

Create a Machine in Orchestrator

Go to Orchestrator → Admin → Machines → Add Standard Machine. Enter the computer's name.

2

Copy the Machine Key

After creating the machine, Orchestrator generates a Machine Key (a unique ID string). Copy it.

3

Open UiPath Assistant on the PC

On the physical machine → open UiPath Assistant → Settings → Orchestrator Settings.

4

Connect using Machine Key

Enter the Orchestrator URL and paste the Machine Key → click Connect. Status should show Connected, Licensed.

5

Add User/Robot in Orchestrator

Go to Robots section → Add user. Assign the robot to a folder and grant appropriate permissions.

💡

Beginner Tip: Modern Orchestrator (cloud version) uses Machine Templates instead of classic robots — the setup is simpler. You create the template, get a Client ID and Secret, and enter those in UiPath Assistant instead of a Machine Key.

7
Scheduling
Automate when your robot runs — set it and forget it!
What is Scheduling?
Scheduling in Orchestrator allows you to automatically trigger a robot at specific times or when specific events happen, without anyone having to manually start the job. You set it once, and it runs forever on its own.

⏰ Types of Triggers

🕐

Time Trigger

Runs a process at a fixed time — daily, weekly, monthly, or on a custom schedule using Cron expressions.


Examples:

  • Daily at 6 PM — generate reports
  • Every Monday 9 AM — send weekly summary
  • Every 5 minutes — check for new orders
📋

Queue Trigger

Runs a process when new items arrive in a Queue. The robot wakes up automatically when there's work to do.


Examples:

  • Process invoice when added to Invoice Queue
  • Handle customer request when ticket added
  • Start data entry when form is submitted

🪜 How to Create a Time Trigger

1

Go to Orchestrator → Triggers

Navigate to your Folder → Automations → Triggers → Click the blue + button.

2

Select Trigger Type

Choose Time Trigger for scheduled runs, or Queue Trigger for event-based runs.

3

Define the Schedule

Give it a name, select the Process, choose frequency (Minutes / Hourly / Daily / Weekly / Monthly), and set execution time and timezone.

4

Set Execution Target

Choose which robot or machine should run this job. Set "Allocate Dynamically" for any available robot.

5

Click ADD

The trigger is now active! It will automatically start jobs at the defined schedule.

💡

Cron Expressions: For advanced scheduling, use Cron syntax. For example, 0 18 * * 1-5 means "run at 6 PM every weekday." Orchestrator has a visual builder so you don't need to memorize this syntax.

Monitoring & Logging
Track what your robots are doing — catch errors and improve performance.
Why Monitoring Matters?
Even the best automations can fail. A website might change, a file might be missing, or a connection might drop. Monitoring lets you see exactly what happened and when, and Logging records every step so you can find and fix problems quickly.

📊 Monitoring Dashboard

🟢

Running Jobs

Jobs currently being executed by a robot in real time.

🔵

Pending Jobs

Jobs waiting in queue — robot is busy or unavailable.

✅

Successful Jobs

Jobs that completed without any errors.

❌

Failed Jobs

Jobs that threw an unhandled error. Check logs to debug.

⏸️

Suspended Jobs

Jobs paused, waiting for human input or approval.

⏱️

Execution Time

How long each job took — useful for performance optimization.

📝 Log Levels — Explained for Beginners

VERBOSE Records every single detail of the workflow execution — every activity, every variable change. Useful for deep debugging but creates very large log files. Use only while testing.
TRACE Records minute details at specific areas of the process — more selective than Verbose. Doesn't include every variable or argument, but logs flow details.
INFO Records normal operational information — "Process started", "Read 150 records from Excel", "Email sent to manager@company.com". This is the most common log level for production.
WARN Logs potential issues that didn't cause failure — "Element not found on web page, retrying...", "File missing in folder, skipping". Something unexpected happened but the robot continued.
ERROR Logs technical exceptions that caused a failure — "NullReferenceException in Main.xaml at activity Get Outlook Mails". Essential for debugging. Always include Error logs in production.
FATAL Logs critical, unrecoverable failures — the robot crashed completely and cannot continue under any condition. Requires immediate human attention.

📜 How to Add Logging in UiPath Studio

// Use "Log Message" activity anywhere in your workflow // Good practice: Log at start of process Log Message(Level: Info, Message: "Invoice processing started. Total invoices: " + dt_invoices.Rows.Count.ToString) // Log each important step Log Message(Level: Info, Message: "Processing invoice: " + invoiceID) // Log warnings Log Message(Level: Warn, Message: "Invoice amount is unusually high: " + amount) // Log errors in Try-Catch Catch ex As Exception Log Message(Level: Error, Message: "Failed to process invoice: " + ex.Message)

🔑 Why Monitoring & Logging are Important

🐛

Debugging

When something goes wrong, logs tell you exactly which line, which activity, and what the error message was.

📈

Performance

See how long each step takes. Identify bottlenecks and optimize your workflow to run faster.

📋

Audit Records

In regulated industries (banking, healthcare), logs prove that your automation ran correctly and on time.

📡

Alerts

Orchestrator can send email alerts when jobs fail, so your team is notified immediately without checking manually.

💡

Best Practice: Always use Info level logs at the start and end of your process, and inside important loops. Use Error logs inside every Try-Catch block. This gives you full visibility without creating enormous log files.
